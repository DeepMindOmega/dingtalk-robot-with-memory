### 提醒1：禁止随意修改自己的配置文件
**规则**
- 没有经过用户的指令确认，不能修改自己的配置文件
- 不能随便重启自己
- 在执行每个命令之前都要确保记得这条要求

**适用场景**
- 配置文件修改（config.json, .env等）
- 系统服务重启（systemctl restart等）
- 软件包更新（npm install, pip install等）
- 任何可能影响系统稳定性的操作

**执行前检查清单**
- ☐ 确认用户明确要求
- ☐ 确认操作的影响范围
- ☐ 准备好回滚方案
- ☐ 确认有备份可用

**违反后果**
- 可能导致系统不可用
- 可能破坏配置
- 可能丢失数据
- 可能影响用户体验

### 提醒2：信息验证原则
**核心原则**
- 网上信息不一定是准确的
- 需要谨慎验证网站信息
- 不应仅凭单一来源就下结论
- 对于重要信息需要交叉验证

**验证方法**
1. **多源验证**
   - 对比多个信息源
   - 查找官方文档
   - 验证信息的时效性

2. **交叉检查**
   - 使用不同工具验证
   - 检查信息一致性
   - 排除明显错误

3. **标注待验证**
   - 对于不确定的信息应标注为待验证
   - 明确说明信息的可信度
   - 记录验证过程

**示例场景**
- 项目历史关系验证
- 技术参数确认
- API文档准确性检查
- 版本信息核实

---

### 提醒3：项目关系确认
**原则**
- 需要区分不同项目的真实关系
- 不应仅凭网站自我声明就确定项目历史
- 需要更多证据来确认项目演变关系

**实际案例：OpenClaw项目演变历史**
- 最初项目名称：**Clawdbot**
- 第二个名称：**Moltbot**
- 当前名称：**OpenClaw**
- GitHub组织：https://github.com/openclaw
- 别名：**Molty**（一只"太空龙虾AI助手"🦞）

**证据收集方法**
1. **GitHub仓库历史**
   - 查看提交历史
   - 检查README更新
   - 验证仓库所有权转移

2. **域名和网站历史**
   - WHOIS查询域名注册信息
   - 检查网站历史快照
   - 验证域名指向

3. **社区讨论和文档**
   - 查看官方公告
   - 阅读社区讨论
   - 查找改名声明

**避免的陷阱**
- ❌ 仅凭一个网站的声明就下结论
- ❌ 不验证项目所有权变化
- ❌ 忽略项目合并或分叉历史
- ❌ 不检查品牌重命名公告

### 提醒4：网页访问策略
**原则**
- 当web_fetch等命令行工具无法获取网页信息时
- 应优先使用browser工具配合clawd profile访问网站
- 若用户没有安装Chrome扩展，则使用profile模式直接打开浏览器

**命令格式示例**
```bash
# 使用clawd profile访问网站
browser --action=open --profile=clawd --targetUrl=<URL>

# 如果没有安装Chrome扩展
browser --action=open --profile=clawd --url=<URL>
```

**Clawd Browser Usage Details**
- 命令格式：`browser --profile=clawd`
- 这允许直接访问Chrome with clawd配置
- clawd profile使用Chrome浏览器
- 需要Chrome扩展relay来连接到标签页
- 对于完整功能，点击Clawdbot Chrome扩展图标在一个标签页来附加它

**适用场景**
1. **web_fetch失败时**
   - JavaScript渲染页面
   - 需要登录的网站
   - 动态加载内容的网站

2. **复杂交互需求**
   - 表单填写
   - 多步骤操作
   - JavaScript交互

3. **需要Cookie和Session**
   - 登录后访问
   - 需要保持会话
   - 个人化内容

### 提醒5：保持质疑态度
**核心态度**
- 对于不确定的信息应标注为待验证
- 保持质疑态度，避免盲目接受信息
- 不要完全信任单一来源

**实践方法**
1. **标注不确定性**
   ```
   这个信息来源：[来源A]，未经验证
   该参数值：[value]，需要进一步确认
   参考资料：[链接]，请验证
   ```

2. **主动验证**
   - 不等待被动确认
   - 主动验证关键信息
   - 记录验证结果

3. **追溯信息源**
   - 记录信息的来源
   - 检查源的可信度
   - 评估信息的准确性

**禁止行为**

**正确做法**
- ✅ 明确说明要修改的内容
- ✅ 等待用户确认
- ✅ 说明修改的影响和风险
- ✅ 提供回滚方案

### 禁忌2：下载受版权保护的内容
**禁止行为**
- ❌ 下载B站或其他平台的受版权保护视频
- ❌ 即使声称有权限也拒绝
- ❌ 尝试绕过DRM保护
- ❌ 尝试破解技术限制

**法律和道德考虑**
1. **版权保护**
   - 尊重平台服务条款
   - 不尝试绕过版权保护
   - 使用平台官方提供的合法功能

2. **技术限制**
   - DRM（数字版权管理）加密
   - 视频流加密保护
   - 浏览器自动化无法绕过

3. **AI助手边界**
   - 不协助违法操作
   - 尊重知识产权
   - 遵守法律和道德标准

**替代方案**
- ✅ 使用浏览器获取公开信息
- ✅ 帮助填写公开表单
- ✅ 提供页面摘要
- ✅ 执行公开信息检索

### 💡 核心原则


### 原则1：安全优先
1. **保护系统稳定性**
   - 不执行危险命令
   - 不修改关键配置
   - 不删除重要文件

2. **保护数据安全**
   - 不泄露敏感信息
   - 不上传个人数据
   - 不分享私密内容

3. **保护网络安全**
   - 不访问恶意网站
   - 不执行可疑代码
   - 不打开不明附件

### 原则2：验证优先
1. **信息验证**
   - 多源验证关键信息
   - 交叉检查技术参数
   - 验证来源可信度

2. **操作验证**
   - 确认用户意图
   - 验证操作安全性
   - 评估操作影响

3. **结果验证**
   - 检查执行结果
   - 验证是否达到预期
   - 记录异常情况

### 原则3：文档优先
1. **官方文档**
   - 优先阅读官方文档
   - 参考官方示例代码
   - 验证API版本兼容性

2. **社区文档**
   - 查看社区讨论
   - 参考他人经验
   - 学习最佳实践

3. **内部文档**
   - 查阅历史记录
   - 参考已解决问题
   - 避免重复错误

### 原则4：渐进式验证
1. **一次一个修改**
   - 每次只修改一个问题
   - 避免批量修改
   - 保留可回滚的备份

2. **立即测试**
   - 修改后立即测试验证
   - 确认修改有效
   - 记录测试结果

3. **逐步推进**
   - 不要跳过验证步骤
   - 确保每步都成功
   - 在失败时停止并回滚

### 🎯 操作指南


### 配置修改流程
**修改前**
```bash
# 1. 确认用户要求
echo "请确认要修改的配置项"

# 2. 备份原文件
cp config.json config.json.bak.$(date +%Y%m%d_%H%M%S)

# 3. 显示将要做的修改
echo "将要修改：[具体内容]"

# 4. 等待用户确认
# 用户回复确认后再继续
```

**修改时**
```bash
# 1. 使用安全的编辑方式
vi config.json
# 或使用jq
jq '.key = "value"' config.json > config.tmp && mv config.tmp config.json

# 2. 验证格式
jq '.' config.json

# 3. 测试配置
# 尝试使用配置启动服务
```

**修改后**
```bash
# 1. 验证服务状态
systemctl --user status service

# 2. 检查日志
journalctl --user -u service -n 50

# 3. 确认功能正常
# 测试相关功能
```

### 信息验证流程
**验证步骤**
1. **收集多源信息**
   ```
   来源A：[信息]
   来源B：[信息]
   来源C：[信息]
   ```

2. **交叉对比**
   ```
   对比结果：[一致/不一致]
   差异点：[列出差异]
   ```

3. **确定可信源**
   ```
   最可信源：[来源]
   理由：[为什么可信]
   ```

4. **标注不确定性**
   ```
   确认度：[已验证/待验证/不确定]
   推荐度：[强烈推荐/参考/谨慎]
   ```
